Update

你提到的路径和文件优先级的规则是Docker Compose的标准行为。当你运行docker compose命令时，Compose 会在当前工作目录中按以下优先顺序寻找配置文件：

  1. compose.yaml（首选）
  2. compose.yml

这两个是推荐的文件命名规则。如果这两个文件都存在，Docker Compose 会优先选择 compose.yaml，因为这是 Docker Compose 规范的最新标准。

此外，为了向后兼容较早版本的 Compose，Compose 还会支持以下文件名：

  1. docker-compose.yaml
  2. docker-compose.yml

如果你的项目中同时存在 compose.yaml 和 docker-compose.yaml，系统也会优先选择 compose.yaml，因为它是最新的规范。

这个机制确保了项目的配置文件可以兼容不同版本的 Docker Compose，同时遵循最新的命名规范。

scoped

在vue中，组件的渲染和scoped样式的绑定顺序是先渲染子组件，然后再进行样式绑定

  1. 子组件的渲染

    1. vue中的渲染时自上而下的，遇到子组件时父组件会有先渲染子组件，然后继续渲染，在渲染时生成的父组件和子组件的标识符是不同的，父组件的DOM节点会带有父组件的data-v-xxxx属性标识符，子组件的DOM节点会带有子组件的data-v-yyyy属性标识符。

    2. 子组件渲染时会生成对应的虚拟的DOM,并挂载到父组件的模板中

  2. scoped样式的生成和绑定       

    1. 当组件的DOM结构生成后，Vue会为该组件的元素添加特定的data属性标识符（例如 data-v-xxxx），以区分不同组件的DOM元素。

    2. 在这个过程中，Vue的scoped机制会确保样式选择器也自动带有这个标识符，只作用于具有相同标识符的元素

slot

概述

  • 组件接受任意类型的js的值作为props，但在某些场景中，需要为子组件传递一些模板片段，让子组件在他们的组件渲染片段

  • 将功能相似和样式相似的组件抽象出带插槽的组件，使用时，将独立部分填充到组建的插槽（Slot)

  • 传入内容不限制是文本，允许传入组件，或者多个元素

  • 父组件显示对插槽传值会覆盖掉插槽初始值

    优势

    1. 灵活：允许在父组件中修改子组件的内容，实现子组件在上下文的复用不需要内部修改

    2. 减少父子组件之间的耦合

    3. 创建更少的组件实现功能，降低开发成本，无需重复创建相似的组件
